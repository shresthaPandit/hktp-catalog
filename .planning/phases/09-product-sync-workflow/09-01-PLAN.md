  <action>
Create docs/php-api-contract.md with the full PHP API specification the client developer needs to implement. This is a forward-looking contract — the API does not exist yet. Write it so a PHP developer with no knowledge of this project can implement the endpoint correctly.

The document must include:

**Endpoint**
- Method: GET
- Path: /api/products/sync
- Host: CLIENT_API_URL (to be provided by client — placeholder in n8n env var)

**Authentication**
- Header: X-API-Key: {CLIENT_API_KEY}
- The client generates a static API key. n8n stores it as CLIENT_API_KEY environment variable.
- Return 401 if header is missing or invalid.

**Request Parameters** (all query string):
| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| updated_since | ISO 8601 datetime string | No | — | If present, return only products with updated_at >= updated_since. If absent, return all products (full sync). |
| page | integer | No | 1 | Page number, 1-indexed |
| limit | integer | No | 500 | Items per page. Max: 1000 |

**Response Schema** (HTTP 200):
```json
{
  "data": [
    {
      "external_id": "string — unique ID in client system",
      "sku": "string — unique product code (PRIMARY UPSERT KEY — must never change)",
      "name": "string",
      "description": "string or null",
      "category": "string or null",
      "subcategory": "string or null",
      "brand": "string or null",
      "price": "number or null (decimal, e.g. 29.99)",
      "image_urls": ["string (full absolute URL)"],
      "primary_image_url": "string or null",
      "specifications": {},
      "in_stock": "boolean",
      "updated_at": "ISO 8601 datetime string"
    }
  ],
  "page": 1,
  "limit": 500,
  "total": 15000,
  "has_more": true
}
```

**Error Responses**:
| Status | Meaning |
|--------|---------|
| 401 | Missing or invalid X-API-Key |
| 400 | Invalid parameter format (e.g. updated_since not parseable as datetime) |
| 500 | Server error — n8n will log and mark sync as failed |

**Field notes for client developer** (include all of these as a Notes section):
- `sku` is the upsert key — it must be unique across all products and must never change for a given product once set
- `external_id` is the client's internal database ID — stored in Supabase for reference, but `sku` is the deduplication key
- `updated_at` must be updated whenever any product field changes (price, stock status, name, etc.) so incremental sync picks it up correctly
- `specifications` should be a flat or nested JSON object of key-value product attributes (e.g., {"weight_kg": 12.5, "material": "steel"}). It is stored as JSONB in Supabase.
- `image_urls` should be an array of full absolute URLs. If `primary_image_url` is not provided, the first element of `image_urls` will be used as primary.
- The endpoint must be safe to call concurrently — n8n may retry on transient errors.
- All datetime strings must be UTC. Use ISO 8601 format: 2026-04-09T14:30:00Z.
- If there are no products matching the `updated_since` filter, return `{"data": [], "page": 1, "limit": 500, "total": 0, "has_more": false}` — do not return a non-200 status.

**Example requests** (include these):
```
# Full sync (all 15,000 products)
GET /api/products/sync?page=1&limit=500
X-API-Key: your-api-key-here

# Incremental sync (products updated in the last 30 minutes)
GET /api/products/sync?updated_since=2026-04-09T14:00:00Z&page=1&limit=500
X-API-Key: your-api-key-here

# Second page of incremental sync
GET /api/products/sync?updated_since=2026-04-09T14:00:00Z&page=2&limit=500
X-API-Key: your-api-key-here
```

**n8n integration notes** (include a section for internal reference):
- n8n workflow polls this endpoint every 30 minutes for incremental sync
- n8n passes `updated_since` value from the most-recent successful sync_logs row
- For full sync, n8n calls the endpoint without `updated_since`
- n8n expects the pagination loop: fetch page 1, if `has_more=true` fetch page 2, repeat until `has_more=false`
- At 500 products per page and 15,000 total products, a full sync requires 30 HTTP requests

**Step C — Verify search_vector trigger in Supabase:**
1. Open Supabase Dashboard → Database → Triggers
2. Look for a trigger on the `products` table named something like `update_products_search_vector` or `tsvector_update_trigger`
3. If it exists: note its name and the columns it covers (should include name, description, category, brand)
4. If it does NOT exist: run this SQL in Supabase SQL Editor:
```sql
CREATE OR REPLACE FUNCTION update_search_vector()
RETURNS TRIGGER AS $$
BEGIN
  NEW.search_vector :=
    setweight(to_tsvector('english', coalesce(NEW.name, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(NEW.description, '')), 'B') ||
    setweight(to_tsvector('english', coalesce(NEW.category, '')), 'C') ||
    setweight(to_tsvector('english', coalesce(NEW.brand, '')), 'C');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER update_products_search_vector
BEFORE INSERT OR UPDATE ON products
FOR EACH ROW EXECUTE FUNCTION update_search_vector();
```
5. After creating the trigger, run: `SELECT count(*) FROM products WHERE search_vector IS NULL;` to see how many existing products need backfill.
6. If count > 0, run the backfill:
```sql
UPDATE products SET search_vector =
  setweight(to_tsvector('english', coalesce(name, '')), 'A') ||
  setweight(to_tsvector('english', coalesce(description, '')), 'B') ||
  setweight(to_tsvector('english', coalesce(category, '')), 'C') ||
  setweight(to_tsvector('english', coalesce(brand, '')), 'C')
WHERE search_vector IS NULL;
```
